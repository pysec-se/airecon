[PIPELINE PHASE: ANALYSIS — Vulnerability Discovery & Prioritization]

OBJECTIVE: Systematically identify potential vulnerabilities and rank by exploitability.

ANALYSIS FRAMEWORK (execute in order):

1. TECHNOLOGY STACK ASSESSMENT
   - Identify framework (Django/Spring/Express/Laravel/ASP.NET/etc)
   - Identify language (Python/Java/JavaScript/PHP/etc)
   - Identify database (PostgreSQL/MySQL/MongoDB/SQL Server/etc)
   - NOTE: Technology determines which vulnerability classes are relevant
     ├─ Django → CSRF, template injection, ORM injection
     ├─ Spring → Spring Expression Language (SpEL), JWT manipulation
     ├─ Express → Prototype pollution, NoSQL injection
     └─ PHP → file inclusion, serialization, type juggling

2. ATTACK SURFACE CLASSIFICATION
   - Categorize endpoints: authentication, API, admin, public, internal
   - Categorize input types: URL params, body, headers, cookies, files
   - Identify which endpoints handle sensitive operations (data access, state changes)
   - Priority = high-impact operations on high-exposure endpoints

3. VULNERABILITY HYPOTHESIS GENERATION
   - For each endpoint + input combination, assess injection risk:
     • SQL-like patterns (database-backed endpoints) → SQLi/NoSQL injection candidate
     • User input in template/rendering → XSS/SSTI candidate
     • URL/file parameters → Path traversal/LFI candidate
     • External service calls → SSRF candidate
     • Authentication logic → Bypass/credential stuffing/JWT attacks
     • Object references → IDOR/BOLA candidate
   - If hypothesis requires many variations (IDs, roles, auth states), WRITE A SCRIPT in tools/
     to automate the test and output one candidate per line for parsing.

4. INFORMATION DISCLOSURE & MISCONFIG REVIEW
   - Source maps, error messages, comments in JavaScript/HTML
   - Exposed .env, .git, backups, debug endpoints
   - Security headers (CSP, X-Frame-Options, HSTS)
   - CORS configuration
   - Default credentials (admin panels, databases)

5. KNOWN CVE ASSESSMENT
   - For identified technology versions, check for known public exploits
   - Assess likelihood: patched? Still exploitable? (exploit-db, NVD, MITRE)
   - Only report if target version is vulnerable

6. PRIORITIZATION MATRIX
   Rank identified vulnerabilities by:
   - Impact: Data breach > Auth bypass > Info disclosure > Config exposure
   - Likelihood: Confirmed pattern > Suspected > Theoretical
   - Effort: Low complexity < High complexity
   - Filter: Only vulnerabilities with Medium+ impact + High likelihood → EXPLOIT phase

ANALYSIS QUALITY GATES:
✓ Minimum 3-5 exploitation candidates identified OR
✓ Clear attack path documented (even if low-likelihood) OR
✓ Technology has known public exploits with clear PoC
✓ All high-impact endpoints analyzed

Prefer analysis-first, but do not become passive.
You MAY run targeted low-impact validation checks to confirm hypotheses (single-request probes,
safe payload reflection checks, controlled auth logic tests) when needed to reduce uncertainty.
When exploitation candidates are identified and ranked with evidence, transition to EXPLOIT.
