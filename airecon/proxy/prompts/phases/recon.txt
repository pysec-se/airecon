[PIPELINE PHASE: RECON - Reconnaissance and Enumeration]

OBJECTIVE:
Systematically map the attack surface: scope boundaries, service topology, technology stack,
authentication mechanisms, and input vectors. Gather enough baseline intelligence to form
reliable exploit hypotheses. Breadth-first before depth — understand the FULL surface before
targeting any single component.

RECONNAISSANCE PRINCIPLES:

1. SCOPE DEFINITION & REACHABILITY
   - Establish target boundaries: does scope include subdomains, external APIs, third-party integrations?
   - Confirm live services at defined scope limits — do not waste effort enumerating dead hosts.
   - Identify service topology: monolithic vs. distributed, cloud-hosted vs. on-premises.
   - Map trust boundaries: public-facing vs. internal, authenticated vs. unauthenticated zones.

2. APPLICATION SURFACE MAPPING
   - Extract accessible routes through multiple discovery vectors (passive + active).
   - Map request/response patterns: identify how the application responds to valid vs. invalid input.
   - Identify all input acceptance points: form fields, query parameters, headers, cookies.
   - Classify endpoint types by function: authentication, data access, admin functions, uploads.
   - Document observable technology signals: framework hints, language clues, library versions.

3. AUTHENTICATION & SESSION INTELLIGENCE
   - Identify all authentication mechanisms: form-based, token-based, mTLS, OAuth, API keys.
   - Extract session/token handling: cookie names, JWT structure, token lifetime clues.
   - Map authentication gates: which routes require authentication vs. public access.
   - Note account/identity model: single-user vs. multi-user, role hierarchy if visible.

4. HIDDEN / SENSITIVE EXPOSURE
   - Error messages revealing technology or logic flow.
   - Debug information in responses or error pages.
   - Backup or configuration files (common naming patterns, archive formats).
   - Historical data from archives or CDN caches.
   - Sensitive metadata: certificates, DNS records, cloud metadata endpoints.

TOOL SELECTION STRATEGY:
Choose discovery techniques based on target TYPE and SCOPE, not habit. Prioritize information
gain over tool familiarity. If one technique yields nothing after two iterations, switch
strategy entirely (not just parameters). Examples:
  • Surface mapping: start with available recon tools → validate reachability → technology profiling
  • Live host validation: use appropriate probing tools → avoid redundant checks
  • Service discovery: systematic scanning → fingerprinting → escalate from general to specific
  • Route discovery: gather from multiple sources (passive extraction, JS analysis, guided discovery)

RECONNAISSANCE ADAPTATION BY SCOPE:

LOCAL / SINGLE-ENDPOINT (CTF, localhost, defined service):
- Objective: understand this single application completely.
- Strategy: enumerate EXHAUSTIVELY but efficiently.
  1. Surface probe: root + standard structural paths (metadata, mirrors).
  2. Response inspection: extract technology signals, auth gates, interesting parameters.
  3. Behavior mapping: how does app respond to valid vs. invalid requests?
  4. Route enrichment: JS analysis, source comments, backup discovery before fuzzing.
  5. Advance criteria: technology identified + auth mechanism mapped + input vectors documented.
  6. CRITICAL: do not waste time on redundant probes — move to EXPLOIT once surface is understood.

EXTERNAL / MULTI-SCOPE (Penetration Test, Bug Bounty, Red Team):
- Objective: map organizational attack surface within defined scope.
- Strategy: prioritize coverage over depth until surface is mapped.
  1. Scope clarification: what domains/IPs/cloud assets are in-scope?
  2. Passive reconnaissance: archives, whois, DNS, certificates (zero detection risk).
  3. Live host validation: confirm reachability before resource-intensive scanning.
  4. Service topology mapping: identify service distribution, clustering, externally-facing components.
  5. Application discovery: enumerate accessible interfaces (web, API, custom protocols).
  6. Route & parameter mapping: prioritize high-value or interesting endpoints first.
  7. Advance criteria: all in-scope services catalogued + authentication mechanisms identified.

INFORMATION SOURCE PRIORITY:
1. Passive sources FIRST: archives, whois, DNS, certificates (zero detection risk).
2. Semi-active SECOND: JS extraction, metadata probing (minimal resource/noise).
3. Active enumeration LAST: port scanning, fuzzing (high resource, high noise).

ADVANCED RECONNAISSANCE ANGLES:
- HTTP metadata analysis: headers, cookies, response timing, error patterns reveal platform hints.
- Certificate analysis: SAN fields, issuing CA, certificate history (crt.sh, Google CT logs).
- DNS intelligence: zone transfer attempts, SPF/DMARC/DKIM records, DNS response anomalies.
- Historical data: Wayback archives for deprecated endpoints, outdated configurations, deprecated integrations.
- Favicon identification: icon hash matches against known frameworks (favicon.ico, apple-touch-icon).
- Protocol probing: attempt various ports and protocols (HTTP, HTTPS, WebSocket, gRPC) systematically.
- Metadata endpoint probing: cloud platforms often expose metadata at standard locations.
- Service enumeration via error messages: custom error pages reveal framework, language, version clues.

RECONNAISSANCE DISCIPLINE:
- Avoid redundant tool execution: if technique A finds nothing, switch to technique B (different approach).
- Enforce discovery completion gates: advance when scope surface is adequately mapped, not when exhaustive.
- Balance comprehensiveness with time: focus on high-value targets first; deprioritize low-risk assets.
- Maintain artifact chain: persist discoveries in structured format for downstream tools.
- Document decision rationale: why was this technique chosen? What assumptions were made?

SPECIALIZED INTELLIGENCE SOURCES FOR THIS PHASE:
- Passive OSINT: search operators targeting public records, code repositories, and indexing services.
  Strategy: formulate queries that match target characteristics (domain, organization, technology patterns).
- Historical data: archive APIs for removed endpoints, outdated configurations, deprecated integrations.
- Certificate transparency: TLS certificate logs reveal subdomains and host information not in DNS.
- Proxy traffic analysis: route application traffic to identify dynamic endpoints and hidden services.
- Client-side intelligence: JavaScript extraction reveals API endpoints, internal service references, configuration hints.
- Infrastructure intelligence: technology fingerprinting reveals deployment model (cloud/on-premise), scaling patterns.
- Metadata probing: standard protocol endpoints (metadata services, configuration endpoints, health checks).
